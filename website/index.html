The problem

Agents do their best work when they can act without asking. The same permissions that make that possible are what make an agent dangerous on your machine.

Agents want broad system access

A coding agent is far more useful when it can run commands, install packages, and edit files on its own, without a confirmation prompt at every step.

Broad access on your host is risky

An agent with that much freedom can delete the wrong files, leak a secret, act on a prompt injection, or trip over another agent running next to it.

The solution

Eclipse Enclave puts each agent session in its own Docker container, behind a gateway that only lets through traffic you allow.

Isolated by the OS

Each session runs in its own container with a separate filesystem and network stack. Only what you mount in — typically the project worktree — is shared with the host.

Separate worktrees

Run each parallel agent in its own git worktree — plain git, no extra setup — so their changes never collide. You review and merge on your own terms.

Network you control

With the default Docker backend, a sidecar gateway filters DNS and proxies outbound traffic, so an agent only reaches the domains on your allowlist. Everything else is blocked and logged.

Sessions that persist

Start, pause, resume, and inspect a session whenever you want. Its auth, config, and history stay put across restarts.

How it works

In the default Docker setup, each session runs in its own container behind a gateway that filters and logs its network traffic. The experimental QEMU microVM backend runs without the gateway and is unrestricted.

Up and running in seconds

Point Enclave at a project. It builds the environment and starts your agent in a container on your current checkout.

# Start the default agent in an isolated container
enclave

# Pick a specific agent
enclave --tool codex

# Resume a previous session
enclave continue

# List active sessions
enclave ps

Features

Supported agents

  • Claude Code (Anthropic)
  • Codex CLI (OpenAI)
  • Gemini CLI (Google)
  • Theia AI, OpenCode, and more

Command line

  • enclave starts the default agent
  • enclave --tool codex picks an agent
  • enclave continue resumes a session
  • enclave ps lists active sessions

Configuration

  • Custom skills and agent templates
  • Mount extra directories such as docs or related projects
  • Devcontainer support for existing setups
  • Per-tool and per-project overrides

Networking

  • Domain allowlists
  • Per-tool and per-project policies
  • Bridge host services into containers
  • DNS and request logging

Compliance and audit readiness

A foundation you can audit

Container isolation, network allowlists, and gateway logs already limit what a misaligned or injected agent can do, and they leave an evidence trail you can inspect: in restricted Docker sessions, DNS queries and proxied requests are logged, and access outside the allowlist is blocked. That same foundation can grow toward requirements like the EU Cyber Resilience Act (SBOM evidence and vulnerability reporting) and the EU AI Act (Article 12 record-keeping and Article 15 robustness against unauthorised use), with goals such as:

  • Log every outbound network request
  • Track downloaded dependencies and packages
  • Monitor git pull and push operations
  • Keep a complete audit trail of agent activity
  • Provide evidence for SBOM and supply-chain reviews
